# BloomCanvas / 生花 代理协作说明

## 项目背景

- 中文名：生花
- 英文名：BloomCanvas
- 仓库名：bloom-canvas
- 产品方向：基于 `gpt-image-2` 及 OpenAI-compatible 图像接口的桌面端 AI 图像生成工作台。
- 首版定位：通用泛化生图，不做项目制；专题能力（Logo、电商主图、详情图等）后续再引入项目制。

## 当前首版范围

首版重点：

- 文生图。
- 参考图生图，参考语义由用户提示词决定。
- 可选提示词优化，不自动覆盖用户原文。
- 本地历史库，记录提示词、参考图、参数、结果图和生成时间。
- 支持自定义 Provider：Base URL、API Key、图像模型、提示词优化模型。
- 默认面向 OpenAI `gpt-image-2`，但不强制使用官方 OpenAI 服务。

首版不做：

- 项目制。
- 局部编辑、蒙版修图。
- 批量生成队列。
- 云同步、账号、额度系统。
- 固定参考图模式（例如强制选择风格/主体/构图）。
- 聊天式多轮会话。

## 技术原则

- 使用 Electron + React + TypeScript。
- UI 主体优先使用 `antd`。
- `@ant-design/x` 只作为提示词输入、附件、建议等局部增强的候选，不作为整体产品骨架。
- Renderer 只负责界面，不直接持有 API Key，不直接请求图像接口。
- Main 进程负责 Provider 调用、凭据读取、文件写入和本地存储。
- API Key 必须保存在系统凭据或加密存储中，不写入普通配置文件。
- Base URL、API Key、模型名都应由用户配置；不要写死官方 OpenAI 地址。

## 数据原则

- 泛化工作台以 `Generation` 作为核心记录，不暴露 Project。
- 图片文件存文件系统，元数据适合放 SQLite。
- 参考图和生成结果统一抽象为 `Asset`。
- 一次生成返回的候选图抽象为 `Variant`。
- 后续专题功能可以再引入 `Project`，但不要影响首版轻量历史库体验。

## 开发习惯

- 修改前先理解现有结构，遵循项目已有风格。
- 文件编辑优先使用 `apply_patch`。
- 不要提交 `.superpowers/`、构建产物、日志、缓存和本地敏感配置。
- 遇到用户已有改动，不要回滚；如果冲突，先说明再处理。
- 代码完成前必须运行合适的验证命令，并在回复中说明验证结果。
