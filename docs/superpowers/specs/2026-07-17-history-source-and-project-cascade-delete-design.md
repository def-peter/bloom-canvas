# 历史来源标识与 Logo 项目级联删除设计

## 背景

当前历史列表混合展示通用创作与 Logo 设计生成记录，用户无法快速判断图片来源。Logo 项目删除按钮在项目仍有生成图片时直接禁用，虽然避免误删，但用户必须先逐张清空图片，操作不直观。

## 目标

1. 每条历史记录明确显示“通用创作”或“Logo 设计”来源标签。
2. 选中 Logo 项目后可直接发起删除，不再要求先手动清空项目图片。
3. 删除确认框明确说明会同步删除项目生成图片和相关历史记录。
4. 用户导入的参考图原文件保留，只解除被删除项目的关联。

## 交互设计

### 历史来源标签

- 历史条目的时间行前增加紧凑标签。
- `generation.scenario === 'logo-design'` 显示“Logo 设计”。
- 其他记录显示“通用创作”，兼容没有 `scenario` 的旧数据。
- 失败记录仍保留失败摘要，来源标签与失败状态同时可见。

### 删除 Logo 项目

- 选中项目且没有生成任务运行时，删除按钮始终可用，不再根据图片数量禁用。
- 确认框展示项目名和当前生成图片数量。
- 有图片时提示：“将同步删除 N 张生成图片和相关历史记录。”
- 无图片时提示：“将删除该项目和相关历史记录。”
- 统一补充：“用户导入的参考图原文件会保留。此操作不可撤销。”
- 生成任务运行期间继续禁用删除，避免删除与写入并发。

## 数据删除边界

主进程以生成记录的 `projectId` 为真实归属，不信任可能过期的 `project.generationIds`。

删除内容：

- Logo 项目记录；
- `projectId` 指向该项目的已完成或失败生成记录；
- 上述生成记录对应的候选图片变体；
- 未被其他生成记录、变体或项目引用的输出资产及缩略图文件。

保留内容：

- 用户导入的参考图资产和文件；
- 其他项目及通用创作的生成记录；
- 被其他创作作为参考图继续使用的输出资产文件；
- 仅因旧 `generationIds` 数据误关联、但 `projectId` 不属于该项目的生成记录。

如项目存在 `pending` 或 `running` 生成记录，删除请求失败且不修改任何元数据。

## 架构

- `HistoryPanel` 只负责把已有 `scenario` 映射为标签，不新增持久化字段。
- `AppShell` 计算当前项目生成图片数量并传给 `LogoProjectPanel`。
- `LogoProjectPanel` 负责删除确认文案和交互状态。
- `LogoProjectService.remove()` 在一次存储更新中完成项目、生成记录、变体和资产元数据清理。
- 资产引用收集与物理文件删除提取为服务层共享工具，供单条生成删除和项目级联删除复用。

## 错误处理

- 删除失败时保留确认框，错误继续由 `AppShell` 展示。
- 文件清理使用强制删除，文件已不存在时不报错。
- 元数据先原子写入，再清理已确定不再引用的物理文件，沿用现有生成删除语义。

## 测试

- 历史列表分别渲染通用创作和 Logo 设计标签。
- 有图片的项目删除按钮可用，确认框包含数量和参考图保留说明。
- 生成中仍禁止删除项目。
- 服务层级联删除项目、生成记录、变体、未复用输出资产和文件。
- 服务层保留参考图、无关生成记录和被其他创作引用的输出资产。
- 完整测试、lint、类型检查和生产构建通过。
